# 长期投资基金日报设计

## 目标

本自动化用于在每个工作日收集半导体与黄金相关信息，并生成适合长期投资者每天阅读的执行型日报。

报告每天只回答三个问题：

- 最近一两个月的信息和基金表现，是否改变长期配置逻辑。
- 今天是否适合执行原定计划。
- 如果执行，具体基金应该正常、减半、小额靠近、暂停，还是只观察。

报告不是新闻列表，也不是盘中喊单。

当前固定覆盖基金：

- `012552`：半导体核心配置。
- `021532`：半导体设备弹性仓。
- `000218`：黄金防守仓。

不再固定覆盖：

- `006533`
- `002611`

## 核心原则

### 长期逻辑优先

长期逻辑复核必须先回答：

- 最近 `1-2` 个月有什么关键原始信息。
- 这些信息是否支持、削弱或改变 `6-24` 个月配置逻辑。
- 实际基金表现是否印证或提醒风险。
- 因此今天的执行节奏如何调整。

### 原始信息必须进入复核，不单独堆放

不再单独渲染“近期信息 / 长期信息”页面区块。

原始信息必须并入对应方向的长期逻辑复核，作为证据支撑。每条证据必须包含：

- 来源。
- 日期。
- 原始事实。
- 链接。
- 对长期逻辑的含义。

这样可以避免报告变成新闻归档。信息只有在支撑判断时才进入主报告。

### 实时数据只是执行过滤器

基金盘中估算、底层代理和黄金 ETF 行情只回答：

- 今天执行原计划是否追高。
- 今天下跌是否只是波动，还是需要暂停。
- 是否需要减半、小额靠近或仅观察。

实时数据不得单独改变长期逻辑。

### 正式净值只做复核

正式净值用于确认前一交易日表现和近 1 日 / 近 1 周 / 近 1 月结果，不得直接替代当天执行过滤器。

## 运行时间

- 频率：每个工作日。
- 建议时间：上午 `10:15-10:30`，默认时区 `Asia/Shanghai`。
- 也允许晚上手动拉取报告，用来结合当日最新数据决定是否行动。
- 若遇中国内地基金市场休市日，仍可生成报告，但必须标注休市，并且不输出当天执行建议。

## 输出文件

每次运行只覆盖：

```text
reports/
  latest/
    data.json
    report.html
    sources.md
```

不保留日期目录。需要追溯时依赖版本控制历史。

## 数据结构

`data.json` 必须先生成，`report.html` 和 `sources.md` 只能从 `data.json` 渲染。

当前数据模型：

```json
{
  "model_version": "long_term_review_v2",
  "report_date": "2026-06-11",
  "generated_at": "2026-06-11T02:30:00.000Z",
  "data_quality": {},
  "top_conclusion": {},
  "thesis_reviews": {},
  "funds": [],
  "source_pool": {},
  "info_source_errors": []
}
```

旧字段 `portfolio_thesis`、`decision_status`、`decision_feed` 不再使用。

### `data_quality`

```json
{
  "status": "complete | partial | blocked",
  "label": "实时执行链路完整",
  "confidence_pct": 100,
  "complete_fund_count": 3,
  "fund_count": 3,
  "oldest_realtime_freshness_minutes": 3.8,
  "can_issue_today_execution": true,
  "reason": "..."
}
```

### `top_conclusion`

```json
{
  "status": "green | yellow | red",
  "status_label": "长期逻辑继续成立",
  "headline": "长期逻辑未变，今天只调整执行节奏",
  "action_summary": "012552 按原计划执行；021532 半额执行；000218 仅观察",
  "reason": "..."
}
```

### `thesis_reviews`

必须包含 `semiconductor` 和 `gold` 两个对象。

```json
{
  "semiconductor": {
    "title": "半导体长期逻辑",
    "status": "green | yellow | red",
    "status_label": "继续成立",
    "conclusion": "...",
    "review_window": "最近 1-2 个月",
    "recent_developments": ["..."],
    "evidence_items": [
      {
        "title": "...",
        "source": "...",
        "time": "...",
        "url": "...",
        "fact": "...",
        "logic_impact": "..."
      }
    ],
    "fund_evidence": [
      {
        "code": "012552",
        "name": "...",
        "role": "...",
        "official_review": "...",
        "intraday_review": "...",
        "interpretation": "..."
      }
    ],
    "allocation_impact": "...",
    "execution_impact": "...",
    "watch_items": ["..."]
  }
}
```

`evidence_items` 必须使用真实原始信息，不允许用基金估值涨跌凑数。

`fund_evidence` 必须使用实际基金数据，不允许泛泛写“走势较强/较弱”。

## 新鲜度规则

每条实时或准实时数据都必须记录：

- `data_time`
- `fetch_time`
- `freshness_minutes`
- `usable_for_today_execution`
- `source_name`
- `source_url`

默认门槛：

- 基金盘中估算：`10` 个有效交易分钟内。
- 底层实时代理：`5` 个有效交易分钟内。
- 黄金 ETF 或黄金现货代理：`5` 个有效交易分钟内。
- 正式净值不限新鲜度，但只能用于复核。

## 基金规则

### `012552`

定位：半导体核心配置。

- 长期逻辑为 `green` 且实时链路完整时，允许按原计划执行。
- 盘中小幅回落且长期逻辑未变时，可以小额靠近。
- 盘中上涨明显时，不追高；可正常执行或减半执行。
- 长期逻辑为 `yellow` 时，只允许减半或观察。
- 长期逻辑为 `red` 时，暂停新增。

### `021532`

定位：半导体设备弹性仓。

- 执行约束必须比 `012552` 更严格。
- 设备链上涨明显时，优先提示不追高和减半。
- 设备链回落时也只能小额，因为波动和回撤都更大。
- 只有设备国产化、行业景气或政策逻辑继续增强时，才允许维持计划内小额执行。

### `000218`

定位：黄金防守仓。

- 黄金不是进攻仓，不因单日下跌自动补仓。
- 若长期防守逻辑未变，优先维持组合比例。
- 只有组合防守仓比例不足，且实时数据没有显示极端失真时，才考虑小额补足。
- 若美元、美债收益率或实际利率信号明显压制黄金，动作偏暂停或观察。

## HTML 输出结构

### 第一层：顶部结论

必须包含：

- 抓取时间。
- 数据质量状态和可信度百分比。
- 长期逻辑总状态。
- 今日动作总建议。
- 一句话原因。

### 第二层：长期逻辑复核

这是报告的核心层，必须替代旧的“原始信息层”。

每个方向必须写：

- 结论。
- 最近 `1-2` 个月发生了什么。
- 原始信息证据。
- 结合基金表现怎么看。
- 是否改变配置逻辑。
- 对今天执行节奏的影响。
- 后续观察项。

证据必须直接展示来源、日期、事实和链接。

### 第三层：今日执行信号汇总

不再写方法论解释。

必须输出具体信号：

- 总体执行结论。
- 触发原因。
- 关键证据。
- 主要风险。

### 第四层：具体基金行动卡

每只基金必须一张行动卡，包含：

- 今日建议。
- 执行幅度。
- 节奏。
- 原因。
- 证据。
- 停止条件。

建议动作只能使用：

- `按原计划执行`
- `半额执行`
- `小额靠近`
- `暂停新增`
- `仅观察`
- `今日不输出执行建议`

### 第五层：正式净值复核

使用表格覆盖三只基金：

- 基金代码。
- 基金名称。
- 基金角色。
- 最新正式净值。
- 近 1 日。
- 近 1 周。
- 近 1 月。
- 复核结论。

正式净值只做复核，不覆盖今日执行结论。

### 第六层：来源与不确定性

必须说明：

- 实时数据来源。
- 正式净值来源。
- 长期复核证据来源。
- 数据缺口。
- 今日建议的主要不确定性。

## `sources.md`

必须记录：

- 实时执行过滤器来源。
- 正式净值复核来源。
- 长期逻辑复核证据来源。
- 数据时间、抓取时间、新鲜度、是否可用于今日执行。
- 可信度备注。

## 写作要求

- 中文撰写。
- 面向非专业投资者，尽量白话。
- 先给结论，再给依据。
- 每段尽量不超过 `120` 个中文字。
- 区分事实、推断和不确定性。
- 不写盘中喊单。
- 不把数据新鲜度写成投资理由。
- 不把长期配置建议和当天执行节奏混为一谈。

## 自动化执行步骤

每次运行必须：

1. 读取本设计文档。
2. 创建或覆盖 `reports/latest/`。
3. 抓取实时执行过滤器数据、正式净值、原始信息证据。
4. 计算新鲜度并生成 `data.json`。
5. 生成 `thesis_reviews`，把原始信息和基金表现合并成长期逻辑复核。
6. 生成每只基金的今日执行计划和停止条件。
7. 生成 `top_conclusion`。
8. 渲染 `report.html`。
9. 渲染 `sources.md`。
10. 自检三份文件是否一致。

## 自检清单

- 是否只覆盖 `012552`、`021532`、`000218`。
- 是否生成 `model_version: long_term_review_v2`。
- 是否不再依赖 `portfolio_thesis`、`decision_status`、`decision_feed`。
- 是否不再单独渲染原始信息层。
- 长期逻辑复核是否包含原始信息证据。
- 长期逻辑复核是否包含实际基金表现证据。
- 今日执行信号汇总是否输出具体判断，而不是方法论说明。
- 每只基金是否有执行幅度和停止条件。
- 正式净值是否只用于复核。
- 数据缺口是否显式披露。
